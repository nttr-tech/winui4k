# COM 参照のライフタイム管理

winui4k のアプリでは、前提の異なる 2 つの寿命管理が同居する。
ネイティブ側の WinUI 3 オブジェクトは COM の参照カウントで寿命が決まり、Kotlin 側のラッパーオブジェクトは JVM のトレース型 GC で寿命が決まる。
本書は、この 2 つを winui4k がどう橋渡ししているかを、COM の参照カウントの基礎から実装の詳細まで順に解説する。

対象読者は、Java または Kotlin の経験があり、COM には馴染みのない開発者を想定する。
レイヤ構成の全体像は `architecture.md` にある。

---

## 第1部 COM の参照カウント

### インターフェースポインタと vtable

**COM** は、Windows がバイナリレベルで定めたオブジェクトの規約である。
WinUI 3 の Button や Window は、言語を問わず同じ形で呼び出せる COM オブジェクトとしてプロセス内に存在する[^winrt]。

[^winrt]: 正確には WinUI 3 のオブジェクトは WinRT オブジェクトである。WinRT は COM の発展形で、共通基底の IInspectable、文字列型 HSTRING、winmd 形式のメタデータを加えたもの。参照管理の規約は COM とまったく同じなので、本書では区別せず COM と呼ぶ。

COM オブジェクトへの参照は、必ず**インターフェースポインタ**の形をとる。
実体は「関数ポインタの配列 (**vtable**) へのポインタを先頭に持つ構造体」へのポインタで、`ポインタ → vtable → vtable[スロット番号]` とたどると目的のメソッドの関数ポインタが得られる。
C++ の仮想関数呼び出しと同じ仕掛けを、言語に依存しない生のメモリレイアウトとして固定したものと考えればよい。

すべてのインターフェースは **IUnknown** を根に持ち、vtable の先頭 3 スロットは必ず次の 3 メソッドになる。

- **QueryInterface** (スロット 0)：同じオブジェクトの別インターフェースを問い合わせる (後述)。
- **AddRef** (スロット 1)：参照カウントを 1 増やす。
- **Release** (スロット 2)：参照カウントを 1 減らす。

### 参照カウントと所有権

COM のオブジェクトは、「自分を参照している者の数」を自分自身で数えている。
これが**参照カウント**で、`AddRef` で増え、`Release` で減り、0 になった瞬間にオブジェクトは自分自身を解放する。
GC のような外部の回収者はいない。
利用者全員が数え上げの規約を守ることだけが、解放漏れと二重解放を防ぐ。

規約の中心にあるのが**所有権**という考え方である。
ポインタを知っていることと、カウント 1 つぶんを所有していることは別で、所有者だけが `Release` の義務を負う。
規約は 3 つに集約できる。

- **戻り値の所有権**：インターフェースポインタを返す関数は、カウントを 1 増やした状態で返す。受け取った側が所有者になり、使い終えたら `Release` する義務を負う。
- **引数の借用**：引数として渡されたポインタは、呼び出しの間だけ借用できる。呼び出しを超えて保持するなら、自分で `AddRef` して所有権を確保する。
- **決定的解放**：最後の `Release` と同時にデストラクタが同期的に実行される。いつ解放されるかをコードで制御できる。

### QueryInterface とビュー

1 つの COM オブジェクトは複数のインターフェースを実装する。
たとえば Button は、ボタン固有の IButton のほかに、XAML ツリー操作用の IUIElement、レイアウト用の IFrameworkElement なども実装している。
別のインターフェースが欲しいときは、**IID** (インターフェースを識別する GUID) を指定して `QueryInterface` (以下 QI) を呼ぶ。

QI の結果は、同じオブジェクトを別の窓から見るポインタである。
本書ではこれを**ビュー**と呼ぶ。
見た目は別のポインタだが、オブジェクトも参照カウントも 1 つしかない。
そして QI の結果も「戻り値の所有権」の規約に従い、カウント 1 増で返る。
つまりビューを 1 本保持するたびに、そのオブジェクトへの参照を 1 カウントぶん所有することになる。

この性質は後の設計に直結する。
ビューはコントロール自身への参照なので、どれか 1 本でも `Release` されずに残れば、カウントは 0 にならず、コントロール本体は解放されない。

### winui4k での最小単位 ComPtr

winui4k がインターフェースポインタを扱う最小単位は `ComPtr` (internal/com) である。
生アドレスを包み、`ポインタ → vtable → vtable[スロット]` の呼び出しと、`getPtr` (出力ポインタを受け取る `HRESULT f(..., T** out)` パターン)、`queryInterface`、`addRef`、`release` を提供する。

`ComPtr` 自体は寿命を管理しない。
`getPtr` や `queryInterface` の戻り値は COM の規約どおり所有権付きで返り、解放は呼び出し側の責任である。
一時的に使うポインタは、winui4k の内部コードでは今も `release()` を明示的に呼んで返している。
問題は、W* ラッパーのように長期間ポインタを保持する場合で、これが本書の主題になる。

## 第2部 トレース型 GC とのミスマッチ

### トレース型 GC の寿命管理

JVM の**トレース型 GC** は、参照の数を数えない。
ルート (スタックや静的フィールド) から参照をたどり、到達できたオブジェクトを生存、できなかったオブジェクトを回収対象とする。

COM との接続で問題になるのは、次の 3 つの性質である。

- 回収のタイミングは非決定的で、到達不能になった瞬間ではなく、次に GC が走ったときにまとめて処理される。
- 回収に伴う後始末は、アプリケーションのスレッドではなく専用スレッドで実行される。
- GC が見ているのは Java ヒープだけで、ラッパーの裏にあるネイティブオブジェクトの大きさは見えない。

### 4 つのミスマッチ

両者を橋渡ししようとすると、4 つのミスマッチに直面する。
以後、本書はこの番号で各問題を参照する。

- **ミスマッチ①(解放通知の不在)**：COM は「不要になったら `Release` を呼べ」と要求するが、GC には「このオブジェクトへの最後の参照が消えた」という瞬間を検出する仕組みがない。
- **ミスマッチ②(ネイティブメモリの不可視)**：数十バイトのラッパーの裏に大きなネイティブオブジェクトがぶら下がっていても、GC はヒープ使用量の変化を観測できず、回収を急ぐ動機を持たない。
- **ミスマッチ③(境界をまたぐ循環)**：ネイティブが Kotlin オブジェクトを参照し、その Kotlin オブジェクトが元のネイティブを参照する循環は、どちらの仕組みも単独では回収できない。
- **ミスマッチ④(解放スレッドの制約)**：COM のオブジェクトには呼び出せるスレッドの制約があるが (第3部で述べる)、GC の後始末は専用スレッドで走る。

### 従来の winui4k の割り切り

自動解放を導入する前の winui4k は、ミスマッチ①を「解放しない」ことで回避していた。
一時的なポインタは手動 `release()` で返す一方、W* ラッパーが保持するコントロール本体の参照は一度も解放せず、プロセス生存期間ぶんのリークを許容していた。
起動して操作して閉じるだけのアプリなら実害はないが、画面を作っては捨てる長時間稼働のアプリでは、捨てたはずのコントロールがネイティブ側に積み上がっていく。

現在の winui4k は、この割り切りを自動解放に置き換えている。
その仕組みが第3部以降の内容である。

## 第3部 自動解放の設計

### ラッパー 1 個に参照カウント 1 つぶんを対応付ける

ミスマッチ①への古典的な解は、「参照カウント 1 つぶんの所有権を、ラッパー 1 個の寿命に対応付ける」ことである。
ラッパーの生成時に確保した参照を、GC がラッパーの到達不能を検出したときに `Release` で返す。
つまり、GC の到達可能性判定を、参照カウントを減らすトリガーとして流用する。
**CsWinRT**[^cswinrt] が C# のファイナライザで行っていることと同じ構図であり、winui4k はこれを JVM の仕組みで実現する。

[^cswinrt]: C#/.NET 向けの WinRT 相互運用ランタイム。COM と .NET GC のあいだで本書と同じ橋渡し問題を解いており、winui4k の設計の下敷きになっている。実装同士の詳細な対比は `cswinrt_vs_winui4k.md` にあるが、本書を読むのに前提とはしない。

このとき、ラッパーとコントロール本体の寿命が別物であることは押さえておきたい。
ラッパーを回収して返すのは「ラッパーが所有していたカウント」だけで、XAML のビジュアルツリーがコントロールに対して持っている参照はそのまま残る。
だから、画面に表示中のコントロールのラッパーが GC されても、コントロールが画面から消えることはない。
逆に、どこにも追加しなかったコントロールは、ラッパーの回収と同時にカウントが 0 になり、ネイティブ側でも破棄される。

### 所有記録 ComLifetime

この対応付けを実装するのが `ComLifetime` (internal/com/lifetime) である。
W* ラッパー 1 個につき 1 個作られ、そのラッパーが所有するすべてのカウントを束ねて記録する。

```kotlin
abstract class WComponent internal constructor(internal val inspectable: ComPtr) {
    private val lifetime = ComLifetime.adopt(this, inspectable)

    internal fun own(ptr: ComPtr): ComPtr = lifetime.own(ptr)

    internal val uiElement: ComPtr by lazy { own(inspectable.queryInterface(XamlInterop.IID_IUIElement)) }
    // ...
}
```

`adopt` は生成時の既定インターフェースの所有権を預かり、`own` は後から QI した各ビューの所有権を追記する。
第1部で述べたとおりビューはコントロール自身への参照なので、長期保持するビューを `own` で包み忘れると、そのラッパーの回収では本体が解放されなくなる。
この規約は第6部で改めて整理する。

### State を分離する理由

`ComLifetime` の内部では、解放に必要な情報 (生ポインタのリストと状態フラグ) を `State` という別オブジェクトに分離し、GC への登録にはそれだけを渡している。
ラッパー本体 (owner) を後始末処理に捕捉させると、登録自体が owner への強参照になり、owner は永遠に到達不能にならないからである。

一方、`ComLifetime` 自身が owner を参照するのは問題ない。
owner → ComLifetime → owner という循環になるが、トレース型 GC は参照カウントと違って循環をまとめて回収できる。
この非対称 (State は owner を参照してはならず、ComLifetime は参照してよい) が、この種の実装でいちばん間違えやすい点である。

### 解放の 2 経路と 3 状態 CAS

解放へ至る経路は 2 つある。
1 つは GC 経路で、owner の到達不能を検出した後始末機構が `State` の処理を呼ぶ。
もう 1 つは明示的な `close()` で、COM 本来の決定的解放に相当する (現時点で W* の公開 API には出しておらず、内部と将来の拡張用である)。

2 つの経路は同時に走りうるため、`State` は `AtomicInteger` の CAS[^cas] で 3 状態を遷移させ、`Release` が一度しか実行されないことを保証する。

[^cas]: compare-and-swap。「現在の値が期待どおりのときだけ新しい値に置き換える」原子操作。複数スレッドが同時に遷移を試みても、成功するのは 1 つだけになる。

- **NOT_DISPOSED**：初期状態。
- **DISPOSE_PENDING**：どちらかの経路が解放処理に入った。
- **DISPOSE_COMPLETED**：解放完了。

`NOT_DISPOSED → DISPOSE_PENDING` の遷移に成功したスレッドだけが解放タスクを発行する。
この 3 状態の設計は CsWinRT から移植したものである。

### Release を UI スレッドへ集約する ReleasePump

ここでミスマッチ④(解放スレッドの制約) を片付ける。
COM には**アパートメント**というスレッド規約がある。
**STA** (single-threaded apartment) に属するオブジェクトのメソッドは、そのアパートメントのスレッドからしか呼べない。
**MTA** (multi-threaded apartment) のオブジェクトは任意のスレッドから呼べ、スレッドを問わない性質は **agile** と呼ばれる。
XAML のオブジェクトは UI スレッド (winui4k では `RoInitialize(STA)` で初期化した 1 本) に束縛されており、`Release` もこの制約の対象になる。

しかし GC の後始末は専用のクリーナースレッドで走る。
そこで winui4k は、クリーナースレッドから直接 `Release` を呼ばず、解放タスクを `ReleasePump` (internal/com/lifetime) へ投函する。
`ReleasePump` は DispatcherQueue 経由でタスクを UI スレッドのメッセージループへ送り、`Release` は常に UI スレッド上で実行される。
UI スレッドが 1 本という winui4k の前提 (`architecture.md` 参照) のもとでは、これでアパートメントの制約を常に満たせる。

この集約には副次的な効果がもう 1 つある。
解放タスクは「現在 UI スレッドで実行中の処理」が終わった後のメッセージループで走るため、UI スレッド上のメソッド実行中にそのオブジェクトが横から解放されることが構造的に起きない。
この性質は第5部の premature finalization の話で効いてくる。

なお、投函の実装 (DispatcherQueue) は winui 層にあり、`ReleasePump` 自体は com 層にある。
レイヤの依存方向 (winui → com) を守るため、投函手段は起動時に winui 層が `ReleasePump.install` で注入する。
Dispatcher の捕捉前に届いたタスクは控えに溜め、捕捉時にまとめて流し込む。

## 第4部 Java バージョンごとの下回り

### 切り替えの全体像

「オブジェクトが到達不能になったら後始末を実行する」ための API は、Java のバージョンで使えるものが異なる。
そこで winui4k は、FFI バックエンド (`Ffi`) と同じ発想で、後始末の下回りも実行時に切り替える。
選択は `Cleanup` (internal/com/lifetime) が行う。

| 実行環境 | 後始末 | fence (第5部) | FFI |
|---|---|---|---|
| Java 8 | `PhantomReference` + 自前デーモンスレッド (phantom) | `synchronized` イディオム | JNA / JNR |
| Java 9〜21 | `java.lang.ref.Cleaner` (cleaner) | `Reference.reachabilityFence` | JNA / JNR |
| Java 22 以上 | cleaner (同上) | `Reference.reachabilityFence` (同上) | Panama (FFM) |

境界が 9 と 22 の 2 箇所にあるのは、層が 2 つあるからである。
寿命管理層の境界は Java 9 (`Cleaner` と `reachabilityFence` の導入) にあり、それより上のバージョン (11、17、21) にはこの層に関係する API の追加がない。
Java 22 の境界は FFM (Foreign Function & Memory API) の正式化によるもので、既存の FFI 層が独立に切り替える。
2 つの層を重ねた結果として、実行環境は表の 3 段に分かれる。

既定はバージョン判定 (`java.specification.version` の解析[^spec-version]) で、システムプロパティ `-Dwinui4k.lifetime=cleaner|phantom` で明示指定もできる。

[^spec-version]: `"1.8"` なら 8、`"9"` 以降はそのまま数値として読む。`Runtime.version()` は Java 9 以降にしか存在しないため使えない。

### Java 9 以上 Cleaner

**Cleaner** (`java.lang.ref.Cleaner`) は、「参照オブジェクトと後始末処理の組」を登録しておくと、参照先が到達不能になったときに専用スレッドで後始末を一度だけ実行してくれる Java 9 の API である。
かつての `finalize()` と違い、オブジェクトの復活がなく、実行は確実に一度きりで、明示解放 (`Cleanable.clean()`) との合流も備えている。
`CleanerCleanupBackend` はこれをほぼそのまま使う。

### Java 8 PhantomReference による自作

Java 8 には Cleaner がないため、`PhantomCleanupBackend` が同じ構造を 3 つの部品で自作する。

**PhantomReference** は「参照先の生存には寄与せず、参照先が回収されたことの通知だけを受け取る」ための参照オブジェクトである。
生成時に **ReferenceQueue** と結び付けておくと、参照先が到達不能になった後、GC がこの参照オブジェクトをキューに入れてくれる。
あとは専用のデーモンスレッドがキューを待ち受け、届いた登録の後始末を実行すればよい。
これは Cleaner 自身の内部実装と同じ構造である。

1 つだけ罠がある。
PhantomReference そのものは普通の Java オブジェクトなので、誰も参照していなければ通知より先に本体が GC されてしまう。
このため生きている登録は集合に入れて強参照で保持し、後始末が済んだら外す。

### なぜ MethodHandle で実行時解決するのか

winui4k のコアモジュールは `-Xjdk-release=8` でコンパイルされ、Java 9 以降の API へのコンパイル時参照を持てない。
それでも `CleanerCleanupBackend` が Cleaner を使えるのは、`Class.forName` と `MethodHandle` でメソッドを実行時に解決しているからである。

同じ制約を持つ FFI 層は、別モジュール (winui4k-ffi-panama) に分離して ServiceLoader で発見する方式をとった。
使い分けの基準は API 面の広さで、FFM のように型が大量に登場する API は別モジュールでの直接参照が読みやすく、Cleaner のようにメソッド 3 つで済む API は実行時解決のほうが配布物を増やさずに済む。

### クリーナースレッドと COM

どちらのバックエンドも、クリーナースレッド (名前は `WinUI4K-Cleaner`) の起動時に `RoInitialize(MTA)` を呼んで COM に参加させる。
後始末は解放タスクを投函するだけだが、その投函 (DispatcherQueue.TryEnqueue) 自体が COM 呼び出しであり、アパートメント未初期化のスレッドからの呼び出しを避けるためである。
DispatcherQueue は agile なので、MTA のスレッドから安全に呼べる。

## 第5部 残りのミスマッチと周辺の機構

### ネイティブメモリの不可視への緩和策

ミスマッチ②に対して、.NET には `GC.AddMemoryPressure` (ネイティブ割り当て量を GC に申告する API) があるが、JVM に対応物はない。
winui4k は近似として `NativeMemoryGovernor` を用意している。
生存中のネイティブ参照の数を数え、`-Dwinui4k.gcThreshold=<参照数>` で指定した閾値を超えるたびに `System.gc()` を要請して、ラッパー回収から `Release` への連鎖を促す。

これはオプトインで、既定では動かない。
参照の「数」しか見ておらず裏のオブジェクトの大きさは相変わらず見えないため、`AddMemoryPressure` の完全な代替ではない。
大きなリソースを短い周期で作る場合の保険と考えるのがよい。

### premature finalization と fence

トレース型 GC には、直感に反する落とし穴がもう 1 つある。
JIT コンパイラは「このオブジェクトのフィールドをもう読まない」と判断した時点で、メソッドの実行中でもオブジェクトを到達不能とみなせる。
ラッパーからポインタを読み出してネイティブ呼び出しを実行している最中に、ラッパー本体が回収されて後始末が走りうる。
これを **premature finalization** と呼ぶ。

対策の道具が **fence** である。
Java 9 以降の `Reference.reachabilityFence(obj)` は「この行まで obj は到達可能」と JIT に保証させる。
Java 8 には API がないため、`synchronized (obj) {}` という既知のイディオムで代用する[^jmm]。
第4部の表の fence 列はこの使い分けを指す。

[^jmm]: 「モニタ操作の対象は生存していなければならない」という Java メモリモデルの性質を利用したもの。

もっとも、winui4k で fence が要る場面は限られている。
第3部で述べたとおり、`Release` は UI スレッドのメッセージループへ集約されるため、UI スレッド上で実行中の W* 呼び出しと衝突することはない (W* API は UI スレッド上でのみ使う契約である)。
fence を実際に使っているのは `ComLifetime.own` で、所有権の登録が後始末より先に完了することを保証している。

### 境界をまたぐ循環の扱い

ミスマッチ③は、winui4k では解決していない。
CsWinRT はこれを .NET GC 自体の拡張点 (`IReferenceTracker` との統合) で解いているが、JVM の GC に相当する拡張点はなく、ライブラリのコードでは実現できない。

具体的に問題になるのは次の形である。
ネイティブのコントロールがイベントハンドラ (Kotlin 実装の COM オブジェクト) への参照を持ち、そのハンドラが Kotlin 側のオブジェクトを捕捉し、その Kotlin オブジェクトが元のコントロールのラッパーを保持する。
この輪の中では、ネイティブ側のカウントも Kotlin 側の到達可能性も互いを支え合い、どちらも 0 にも到達不能にもならない。

回避は利用側の規約になる。
不要になったリスナーは remove 系メソッドで解除する、ラッパーへの参照を捕捉したまま放置しない、という通常のイベント購読の作法がそのまま対策である。

### 逆方向 KComObject の参照カウント

ここまではネイティブのオブジェクトを Kotlin から使う方向 (いわゆる **RCW**、runtime callable wrapper) の話だった。
逆に、イベントハンドラのように Kotlin 側の実装をネイティブへ COM オブジェクトとして見せる方向 (**CCW**、COM callable wrapper) は `KComObject` (internal/winrt) が担う。

`KComObject` は COM 側の規約に合わせて自前の参照カウント (`AtomicInteger`、生成時 1) を実装する。
カウントが 0 でない間は、グローバルな登録表がメソッド実体 (Kotlin のラムダ) を強参照して GC から保護し、ネイティブからの最後の `Release` で登録を外して GC 可能に戻す。
「カウントが生きている間はマネージ側を守る」という不変条件は、.NET ランタイムが CCW に対して行っていることと同じである。

この 2 方向は連動する。
コントロール本体がラッパーの回収で破棄されると、そのイベントテーブルが保持していたハンドラの参照も `Release` され、`KComObject` の登録が外れて Kotlin 側のリスナーも回収される。
自動解放の導入で、CCW 側の回収も実際に機能するようになった。

### プロセス終了時

最後のウィンドウが閉じてメッセージループが終わると、winui4k は `RoUninitialize` で COM アパートメントを閉じる。
その後に `Release` を呼ぶとクラッシュを招くため、`RoUninitialize` の前に `ReleasePump.shutdown()` を呼び、以後の解放要求はすべて破棄する。
終了間際に未解放の参照が残っていても追いかけず、プロセス終了に任せる。
CsWinRT も同じ割り切りをしている。

## 第6部 実践ガイド

### ラッパーを書くときの規約

W* クラスを追加・拡張するときは、次の規約を守る。
どれも第1部の「ビューはコントロール自身への参照」という事実の帰結である。

- QI や `getPtr` で得たポインタをフィールドで長期保持するなら、必ず `own(...)` で包む。包み忘れた 1 カウントが残ると、ラッパーを回収してもコントロール本体は解放されない。
- `QI → getPtr → QI` のようなチェーンでは、中間のポインタも所有権付きで返っている。中間もそれぞれ `own` するか、使い終えた場で `release()` する。
- インスタンス生成は `Activation.activate(cls, iid)` の 2 引数版を使う。`activate(cls).queryInterface(iid)` と書くと中間の参照が 1 カウントリークする。
- 呼び出しの間だけ使う一時的なポインタは、従来どおり明示的に `release()` する。自動解放はフィールド保持のためのもので、一時利用の置き換えではない。
- WComponent 派生でない XAML オブジェクトのラッパーには、`private val lifetime = ComLifetime.adopt(this, inspectable)` を持たせる (WSwipeItems や WTreeNode が例)。
- ウィンドウと Shell 系のラッパー (WFrame、WAppWindow、WAppNotification、WJumpList など) は自動解放の対象外で、従来どおり保持し続ける。

### 動作検証の手法

自動解放は GC のタイミングに依存するため、普通に操作するだけでは経路が通らないことがある。
検証には `NativeMemoryGovernor` を流用して GC を強制的に多発させるのが手軽である。

```
JAVA_TOOL_OPTIONS=-Dwinui4k.gcThreshold=200 ./gradlew :winui4k-sample-gallery:run
```

この状態で Gallery のカテゴリページを何往復か切り替えると、ページの再構築のたびに旧ページのラッパー群が到達不能になり、クリーナー経由の `Release` が UI スレッドで大量に実行される。
切り替え後も UI が操作でき、標準エラーに `[winui4k]` の解放失敗が出ず、ウィンドウを閉じて終了コード 0 で終われば、生成、解放、終了の全経路が通っている。
Java 8 系統は `runJna` タスクで同じ手順を踏めば、phantom バックエンドと JNA の組で確認できる。
